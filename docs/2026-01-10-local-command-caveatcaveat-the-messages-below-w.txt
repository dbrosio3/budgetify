  Context Manager Limitations (context-manager.ts)

  - Stores only the last confirmed transaction (24h TTL)
  - No conversation history
  - No pending question state
  - Key insight: contextManager is really a "last transaction manager", not a conversation context

  What Needs to Change for Session-Based Conversations

  1. Conversation History Storage - Store recent messages (not just last transaction)
  2. AI Response Types - Allow clarifying questions alongside JSON
  3. State Machine - Track if bot is awaiting clarification vs. awaiting confirmation
  4. Prompt Restructuring - Tell AI it CAN ask questions when uncertain
  5. Message Threading - Link follow-up messages to original transaction intent

     # Session-Based Conversation System for Budgetify Bot
     ## Overview
     Transform the bot from a single-message transaction flow to a
     session-based conversation system where:
     - AI can ask clarifying questions when uncertain
     - Users can make clarifications after the initial message
     - Conversation history is maintained within sessions (1h TTL)
     - Sessions persist across restarts via Redis

⏺ I've written the plan to the plan file. Here's a summary of what's proposed:

  Plan Summary

  Goal: Transform from single-message → session-based conversations with AI clarification capabilities

  Key Components

  1. Session Manager (Redis) - Stores conversation state with 1h TTL
  2. State Machine - Tracks idle → processing → awaiting_clarification/confirmation
  3. Discriminated Union AI Response - Transaction, Clarification, or Error
  4. Mixed UI - Buttons for known options, text for open-ended fields

  Implementation Scope

  - 4 new files to create (session-manager, conversation-handler, response-parser, clarification-builder)
  - 7 files to modify (types, prompts, handlers, callbacks, server, config, package.json)
  - 2 files to deprecate eventually (context-manager, pending-operations)

  Infrastructure

  - Adds Redis as a dependency (ioredis)
  - New env variable: REDIS_URL
